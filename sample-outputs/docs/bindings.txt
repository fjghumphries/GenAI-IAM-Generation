================================================================================
                     DYNATRACE IAM POLICY BINDINGS
================================================================================

This document describes how policies are bound to groups.
Bindings connect WHO (groups) to WHAT (policies) and WHERE (boundaries).

IMPORTANT: Bindings for the same group are split into separate
dynatrace_iam_policy_bindings_v2 resources when they use different
boundary types (storage vs settings cannot share the same binding).

================================================================================
SECTION 1: BOUNDARIES REFERENCE
================================================================================

Boundaries define WHERE policies apply. They restrict access based on
dt.security_context using startsWith() for hierarchical matching.

Total boundaries (sample config): 8 (2 BU×2 types + 2 Application×2 types) | At scale: 4,020

--------------------------------------------------------------------------------
1.1 BU Data Boundaries (storage:dt.security_context)
--------------------------------------------------------------------------------
Name: Boundary-BU1
Query: storage:dt.security_context startsWith "BU1-";
Covers: All stages and applications in BU1

Name: Boundary-BU2
Query: storage:dt.security_context startsWith "BU2-";

--------------------------------------------------------------------------------
1.2 BU Settings Boundaries (settings:dt.security_context)
--------------------------------------------------------------------------------
Name: Boundary-BU1-Settings
Query: settings:dt.security_context startsWith "BU1-";
Covers: Settings on entities with BU1 security context

Name: Boundary-BU2-Settings
Query: settings:dt.security_context startsWith "BU2-";

--------------------------------------------------------------------------------
1.3 Application Data Boundaries (storage:dt.security_context)
--------------------------------------------------------------------------------
Name: Boundary-PETCLINIC01
Query:
  storage:dt.security_context startsWith "BU1-PROD-PETCLINIC01";
  storage:dt.security_context startsWith "BU1-DEV-PETCLINIC01";
Covers: All active stages for PetClinic 01 (BU1)

Name: Boundary-PETCLINIC02
Query:
  storage:dt.security_context startsWith "BU2-PROD-PETCLINIC02";
  storage:dt.security_context startsWith "BU2-DEV-PETCLINIC02";
Covers: All active stages for PetClinic 02 (BU2)

--------------------------------------------------------------------------------
1.4 Application Settings Boundaries (settings:dt.security_context)
--------------------------------------------------------------------------------
Name: Boundary-PETCLINIC01-Settings
Query:
  settings:dt.security_context startsWith "BU1-PROD-PETCLINIC01";
  settings:dt.security_context startsWith "BU1-DEV-PETCLINIC01";

Name: Boundary-PETCLINIC02-Settings
Query:
  settings:dt.security_context startsWith "BU2-PROD-PETCLINIC02";
  settings:dt.security_context startsWith "BU2-DEV-PETCLINIC02";


================================================================================
SECTION 2: BU ADMIN BINDINGS
================================================================================

BU Admins use Standard User + Admin Features (custom policy) instead of
Admin User. This ensures settings:objects:write is ONLY granted via the
bounded Scoped Settings Write policy.

Admin User default policy is intentionally NOT used because it grants
unconditional settings:objects:write that cannot be scoped.

--------------------------------------------------------------------------------
2.1 BU1-Admins - Binding Resource 1 (data + features)
--------------------------------------------------------------------------------
┌─────────────────────────────────────────────────────────────────────────────┐
│ Policy: Standard User (default)                                             │
│ Boundary: None                                                              │
│ Result: Documents, Davis AI, segments, SLO read, automation read, etc.      │
├─────────────────────────────────────────────────────────────────────────────┤
│ Policy: Admin Features (custom)                                             │
│ Boundary: None                                                              │
│ Result: Full automation admin, SLO write, extensions write, OpenPipeline,   │
│         App Engine — NO settings:objects:write                               │
├─────────────────────────────────────────────────────────────────────────────┤
│ Policy: Scoped Grail Data Read (templated)                                  │
│ Boundary: Boundary-BU1                                                      │
│ Parameters: security_context_prefix = "BU1-"                                │
│ Result: Read logs, metrics, spans, events, bizevents for all BU1 data       │
├─────────────────────────────────────────────────────────────────────────────┤
│ Policy: Read Entities (default)                                             │
│ Boundary: Boundary-BU1                                                      │
│ Result: Read entity data scoped to BU1                                      │
├─────────────────────────────────────────────────────────────────────────────┤
│ Policy: Read System Events (default)                                        │
│ Boundary: None                                                              │
│ Result: Access to system/audit events (not security_context scoped)         │
└─────────────────────────────────────────────────────────────────────────────┘

--------------------------------------------------------------------------------
2.2 BU1-Admins - Binding Resource 2 (settings)
--------------------------------------------------------------------------------
┌─────────────────────────────────────────────────────────────────────────────┐
│ Policy: Scoped Settings Write (templated)                                   │
│ Boundary: Boundary-BU1-Settings                                             │
│ Parameters: security_context_prefix = "BU1-"                                │
│ Result: Read/write settings scoped to BU1 entities                          │
│ Note: This is the ONLY source of settings:objects:write for BU Admins       │
└─────────────────────────────────────────────────────────────────────────────┘

BU2-Admins: Same structure with Boundary-BU2 and Boundary-BU2-Settings.

├─────────────────────────────────────────────────────────────────────────────┤

│ Boundary: Boundary-BU1                                                      │
│ Result: Read entity data scoped to BU1                                      │

SECTION 6: BINDING SUMMARY MATRIX
================================================================================

┌───────────────────────┬────────────┬───────────────┬──────────────┬─────────┐
│ Group                 │ Base Policy│ Grail Data    │ Settings     │ SLOs    │
│                       │            │ (+ boundary)  │ Write        │         │
├───────────────────────┼────────────┼───────────────┼──────────────┼─────────┤
│ BU1-Admins            │ Std+AdmFt  │ ✓ BU1 scope         │ ✓ BU1 scope  │ ✓ Write │
│ BU1-Users             │ Std User   │ ✓ BU1 scope         │ Read only    │ Read    │
│ BU2-Admins            │ Std+AdmFt  │ ✓ BU2 scope         │ ✓ BU2 scope  │ ✓ Write │
│ BU2-Users             │ Std User   │ ✓ BU2 scope         │ Read only    │ Read    │
├───────────────────────┼────────────┼─────────────────────┼──────────────┼─────────┤
│ PETCLINIC01-Admins    │ Std User   │ ✓ BU1-*-PETCLINIC01 │ ✓ PC01 scope │ ✓ Write │
│ PETCLINIC01-Users     │ Std User   │ ✓ BU1-*-PETCLINIC01 │ Read only    │ Read    │
│ PETCLINIC02-Admins    │ Std User   │ ✓ BU2-*-PETCLINIC02 │ ✓ PC02 scope │ ✓ Write │
│ PETCLINIC02-Users     │ Std User   │ ✓ BU2-*-PETCLINIC02 │ Read only    │ Read    │
└───────────────────────┴────────────┴───────────────┴──────────────┴─────────┘

Std+AdmFt = Standard User + Admin Features (custom policy)

Binding resource count per group type:
  BU Admin:          2 resources (data binding + settings binding)
  BU User:           1 resource  (data + settings read in one binding)
  Application Admin: 2 resources (data binding + settings/SLO binding)
  Application User:  1 resource  (data + settings read in one binding)
